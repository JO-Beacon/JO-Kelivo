## 2. 当前已生效且必须保护的 JO 独立化项目

本节记录当前 1.1.16 主项目中已经恢复并应在后续同步上游时保护的 JO-Kelivo 身份、运行、发布和合规差异。

### 2.1 应用身份与平台隔离

目标：JO-Kelivo 必须能与原版 Kelivo 并存，不能共用平台包名、Bundle ID、应用 ID、二进制名或运行实例标识。

保护点：

- Android `namespace` 和 `applicationId`：`com.psyche.jokelivo`。
- iOS Bundle ID：`com.psyche.jokelivo`。
- macOS Bundle ID：`com.psyche.jokelivo`。
- Windows 可执行文件名和 CMake target：`jo_kelivo`。
- Linux binary name：`jo_kelivo`。
- Linux application ID：`com.psyche.jokelivo`。
- Web manifest、Web title、平台显示名：`JO-Kelivo`。
- 应用图标继续使用 JO-Kelivo 品牌资源。

保护范围：

- `android/app/build.gradle.kts`
- `ios/Runner/Info.plist`
- `macos/Runner/Configs/AppInfo.xcconfig`
- `windows/CMakeLists.txt`
- `windows/runner/Runner.rc`
- `linux/CMakeLists.txt`
- `linux/runner/my_application.cc`
- `web/index.html`
- `web/manifest.json`
- `flutter_launcher_icons.yaml`

同步检查点：

- 不能被上游覆盖回 `com.psyche.kelivo`、`Kelivo` 二进制名或原版 Web 标题。
- 不能为了减少差异把 Dart package name 从 `Kelivo` 改掉；这会破坏现有 `package:Kelivo/...` 导入。
- 平台身份变更后必须做对应平台构建或说明未验证风险。

### 2.2 用户数据目录隔离

目标：JO-Kelivo 与原版 Kelivo 不共用本地数据目录，避免聊天、设置、缓存、上传文件和日志互相污染。

保护点：

- `AppDirectories.appDirName` 必须保持 JO-Kelivo 专属目录名。
- Android 后台服务、通知、备份/恢复和文件路径相关逻辑不得回退到原版目录假设。
- 不能通过静默 fallback 把 JO-Kelivo 数据读写回原版 Kelivo 目录。

保护范围：

- `lib/utils/app_directories.dart`
- `lib/core/services/android_background.dart`
- `lib/core/services/notification_service.dart`
- 备份、导入、上传文件、缓存、日志等路径相关模块。

同步检查点：

- 新增路径逻辑时必须复用 `AppDirectories`，不能散落硬编码平台路径。
- 任何数据迁移都必须显式、有日志、有失败提示，不能静默迁移或假成功。

### 2.3 Windows 单实例隔离

目标：原版 Kelivo 与 JO-Kelivo 同时存在时，Windows 单实例逻辑只能唤起 JO-Kelivo 自己的窗口。

保护点：

- Windows 命名互斥体保持 `JOKelivoMutex`。
- 主窗口类保持 Flutter runner 默认窗口类，不能为了隔离改坏 `bitsdojo_window` 依赖的窗口初始化链路。
- 已有窗口查找必须使用 JO-Kelivo 专属窗口属性过滤，不能只靠窗口标题。

保护范围：

- `windows/runner/main.cpp`
- `windows/runner/win32_window.cpp`
- `windows/runner/win32_window.h`

同步检查点：

- 原版 Kelivo 正在运行时，启动 JO-Kelivo 不能置前原版窗口。
- JO-Kelivo 已运行时，再次启动 JO-Kelivo 应唤起已有 JO-Kelivo 窗口。
- 修改后优先运行 `flutter build windows`；如环境限制无法运行，必须记录风险。

### 2.4 桌面托盘品牌名

目标：桌面托盘悬停提示必须显示 `JO-Kelivo`，不能残留原版 `Kelivo`。

保护点：

- Windows / macOS 托盘 tooltip 使用 `JO-Kelivo`。

### 2.5 更新源与 Release asset 匹配

目标：JO-Kelivo 的新版本检测只面向 JO-Kelivo 自己的 GitHub Releases，不能跟随原版 Kelivo 更新源。

保护点：

- 更新源固定为：`https://api.github.com/repos/JO-Beacon/JO-Kelivo/releases/latest`。
- 版本解析使用 GitHub Release `tag_name`，允许去掉开头 `v` / `V` 后再比较。
- 下载链接从 Release assets 中按当前平台匹配真实安装包或压缩包。
- Release asset 匹配只接受 `JO-Kelivo` 产物名。
- `.sha1` / `.sha256` 校验文件不得作为用户点击下载的安装包。

保护范围：

- `lib/core/providers/update_provider.dart`
- `lib/features/home/widgets/side_drawer.dart`
- `test/update_provider_test.dart`

同步检查点：

- 不能回退到原版 `https://kelivo.psycheas.top/update.json`。
- 不能用原版 `update.json` 格式替代 GitHub Release 格式。
- Android 优先匹配 `arm64-v8a` APK；Windows 优先 `setup.exe`，无安装包时再用 `portable.zip`；Linux 优先 `AppImage`；macOS 使用 `dmg`。

### 2.6 版本号、发布命名和 Release 合规

目标：对外发布必须清晰标明 JO-Kelivo 项目、版本、平台、架构和分发形态。

保护点：

- `pubspec.yaml` 当前版本为 `0.1.3+3`。
- 不直接使用上游 Kelivo 版本号作为 JO-Kelivo 发布版本。
- Release 二进制统一命名：

```text
JO-Kelivo-v{version}-{platform}-{arch}-{package}.{ext}
```

字段规则：

- `version` 来自 `pubspec.yaml`。
- `platform` 固定小写，例如 `android`、`windows`、`macos`、`linux`、`ios`。
- `arch` 表示目标架构，例如 `arm64-v8a`、`armeabi-v7a`、`x86_64`、`x64`、`arm64`、`universal`。
- `package` 表示分发形态，例如 `release`、`setup`、`portable`、`dmg`、`appimage`、`deb`、`rpm`、`archive`。
- 校验文件必须使用完整产物文件名追加算法后缀，例如 `.sha256`。

保护范围：

- `pubspec.yaml`
- `.github/workflows/**`
- `scripts/windows/build_installer.ps1`
- `scripts/windows/kelivo_installer.iss`
- `Release日志.md`
- `README.md`

同步检查点：

- 不能发布模糊文件名，例如 `app-release.apk`、`setup.exe`、`latest.zip`。
- Release notes 必须说明对应上游版本和发布时间。
- 发布二进制时必须同时提供对应源码获取方式。
- README 必须保留“非官方改版”、原版作者致谢和 AGPL-3.0 合规说明。

### 2.11 Linux-only GitHub Actions 构建发布

目标：当前 CI 只保留 Linux x64 自动化打包入口，避免日常推送消耗 Actions 额度或生成临时安装包。

处理结论：

- 已移除旧多平台 workflow，包含损坏并导致 GitHub 报 `Invalid workflow file` 的 `build-stable-44.yml`。
- 当前 Linux 构建入口为 `.github/workflows/build-linux.yml`。
- 当前 PR 检查入口为 `.github/workflows/pr-check.yml`。
- 不再支持 macOS、Windows、Android、iOS 或 Linux ARM64 的自动打包 workflow；如未来恢复，必须作为单独任务重新设计和验证。

触发规则：

- `build-linux.yml` 保留 `workflow_dispatch`，允许维护者在 GitHub Actions 页面手动触发 Linux 构建。
- `build-linux.yml` 只在推送 `v*` tag 时自动触发，例如 `v0.1.4`。
- 日常推送到 `main` 不触发 Linux 打包。
- `build-linux.yml` 不设置 `pull_request` 触发，避免 PR 自动生成安装包。
- `pr-check.yml` 的 PR 目标分支为 `main`，不得回退到 `master`。

Linux 产物：

- `tar.gz`
- `AppImage`
- `deb`

发布方式：

- 普通日常提交：只执行 `git push origin main`，不会触发 Linux 打包。
- 需要发布 Linux 构建时，先创建并推送版本 tag，例如：

```bash
git tag v0.1.4
git push origin v0.1.4
```

- 推送 `v*` tag 后，GitHub Actions 自动运行 `Build Linux`，并上传 Linux x64 产物。
- 手动触发时可选择是否发布 GitHub Release；tag 触发时按 tag 名称发布 Release。

保护范围：

- `.github/workflows/build-linux.yml`
- `.github/workflows/pr-check.yml`
- `.github/workflows/**`

同步检查点：

- 不要恢复损坏或重复的旧多平台 workflow。
- 不要让 `build-linux.yml` 重新监听 `main` 分支 push 或 PR。
- 不要把 PR 检查目标分支改回 `master`。
- 不要在没有明确发布任务时恢复 macOS、Windows、Android、iOS 或 Linux ARM64 打包。
- 修改 workflow 后必须检查所有 active workflow 的触发条件，确认日常 push 不会自动打包。
- 修改 workflow 后至少做 YAML 可解析检查；真正 Linux 打包结果以 GitHub Actions 运行结果为准。

### 3.1 单条聊天记录身份切换

功能目标：用户可在单条消息的更多菜单中，把消息身份在用户和模型之间切换，并真实持久化到聊天记录。

- 只允许 `user` 与 `assistant` 两类切换，不扩展未定义角色。

### 3.2 新建或复制助手插入顶部设置

### 3.3 宽屏聊天区域拉宽设置

功能目标：用户可在显示设置中控制宽屏聊天页是否使用更宽的消息区和输入区。默认关闭，避免改变默认阅读体验。

保护点：

- 设置默认关闭。
- 当前 key：`display_wide_chat_layout_v1`。
- 兼容旧 JO key：`display_desktop_wide_chat_layout_v1`。
- API 使用中性宽屏语义：`wideChatLayout`，不要新增 desktop-only API。

### 4.1 长会话版本消息写入层修复与旧存档优化工具

处理结论：已按融合方案恢复。当前主项目保留上游 Kelivo 1.1.16 的显示层窗口过滤修法，同时恢复旧 JO 1.1.15 的写入层源头修法，并恢复独立旧聊天存档优化工具。

差异结论：

- 上游 Kelivo 1.1.16 仍会把编辑或重新生成产生的新版本消息写到 `Conversation.messageIds` 尾部，但通过 `ChatController._messagesWithVisibleGroups()` 补齐当前窗口涉及的版本组，并用组锚点位置过滤“锚点在当前懒加载窗口之前”的尾部旧版本，避免显示错乱。
- 旧 JO 1.1.15 的修法不同：在 `ChatService.addMessage()` 和 `ChatService.appendMessageVersion()` 写入时，把 `groupId` 已存在且 `version > 0` 的版本消息插回同一版本组附近，从源头避免继续制造尾部错位版本。
- 两种修法不互斥：显示层过滤用于兼容旧脏数据，写入层插回用于阻止新数据继续复发。

保护点：

- `ChatController._messagesWithVisibleGroups()` 的组补全、组锚点过滤和窗口从同组中间开始时的例外逻辑必须保留。
- `ChatService.addMessage()` 对普通新消息继续追加到尾部；只有传入有效 `groupId` 且 `version > 0` 的已有组新版本才插回同组最后一个版本之后。
- `ChatService.appendMessageVersion()` 创建的新版本必须插回原消息组附近，并同步更新内存 cache 顺序和 `versionSelections`。
- 找不到同组锚点时，版本消息保守追加到尾部，不抛错、不伪造锚点。
- 旧聊天存档优化工具只存在于 `optimize_chat_archive/`，作为显式 CLI 工具处理 `chats.json`；不得接入应用启动、打开会话、备份导入或任何静默自动迁移流程。
- 优化工具只调整 conversation 的 `messageIds` 顺序，不修改消息正文、角色、版本号、时间戳、会话 ID、`versionSelections` 或备份 schema。

## 7. 当前重写后的使用方式

后续维护时，本文件按以下规则更新：

1. 新功能或旧补丁只有在当前主项目实际恢复、生成、格式化并完成必要验证后，才能移入“当前已生效且必须保护”章节。
2. 只存在于旧 JO 归档里的功能，必须留在“待回放/待融合”章节。
3. 如果确认某个旧补丁已被上游 1.1.16 原生能力覆盖，应记录为“已由上游覆盖，JO 差异废弃”，不要继续要求保护旧实现。
4. 每次迁移阶段完成后，优先更新 `plans/stage*_record.md`，再同步更新本文件的当前状态。
5. 本文件是保护清单，不是愿望清单；未实现、未验证、未确认的项目不能写成已生效。
